## 2026-05-11 — Supabase SQL Editor + AI-автозаполнение портят `type`
**Контекст:** При вводе SQL в SQL Editor через браузерную автоматизацию (`computer.type`) Monaco-редактор Supabase инжектировал чужие токены прямо в текст — в комментарий вклеилось `grant_pg_cron_access`, ON-блок собрался не так.
**Урок:** Monaco в Supabase активно вмешивается в посимвольный ввод через AI-подсказки и автозаполнение.
**Как избежать:** Для программной вставки больших SQL — `monaco.editor.getEditors()[0].setValue(sql)` через JavaScript, а не `type`. Сразу даёт чистый текст одним блоком, никаких подсказок не успевает сработать.

## 2026-05-11 — Cursor рапортует «закоммитил», не сделав push
**Контекст:** Cursor отчитался «Коммит 58705fc. Всё чисто», но на remote main коммита не было — он остался локально. Quick-fix через GitHub web уехал в прод раньше, и без проверки могли применить SQL-миграцию для несуществующего на проде кода.
**Урок:** Локальный commit ≠ push. Cursor не всегда явно различает в отчёте.
**Как избежать:** После Cursor-сессии с изменениями кода — обязательная проверка `git ls-remote origin main` или через GitHub UI. В промт Cursor стоит включать строку: «после commit выполни git push и подтверди хеш на remote через git ls-remote origin main».

## 2026-05-11 — Привязка промта к файлам ломается, если карта проекта не свежая
**Контекст:** В промте для архитектурного фикса watchdog указан `health_monitor.py` как место APScheduler-джобов (со ссылкой на старый pr.419). Фактически джобы лежат в `main_railway.py`. Cursor сам сообразил и поправил, но мог и слепо последовать.
**Урок:** Указывать в промте конкретный файл рискованно, если ты не видел его прямо сейчас. Старая карта проекта врёт быстрее, чем мы её обновляем.
**Как избежать:** Формулировать через функционал, не через путь: «найди APScheduler-джобы и измени cadence для платных методов» вместо «отредактируй health_monitor.py строка X». Cursor сам найдёт правильное место.

## 2026-05-11 — IDEA: Project Trackers Sheet (отложено)
**Контекст:** При разведке (pr.423) выяснилось, что у проекта нет рабочей таблицы для трекинга. Геннадий подтвердил, что это полезно завести.
**Идея:** Завести Google Sheets с несколькими вкладками:
- **Tasks** — открытые задачи: задача / приоритет / статус / дата создания / связанный коммит или PR / ответственный (Геннадий / Cursor / Claude).
- **Finance** — расходы на API (Anthropic, Groq, RapidAPI, Supadata, Webshare, Railway, Supabase, ЮKassa fees) и доходы по тарифам (Базовый / Стандарт / Про / Overflow), ROI по месяцам.
- **Metrics** — DAU/WAU бота, конверсия из триала в платный, retention, средняя длительность видео — автообновление из Supabase через Apps Script или ручной импорт.
- **Clients/Leads** — если будут B2B-заказчики (Kwork и т.п.).
**Как реализовать:** Создать Google Sheets в Drive (smolnirovgv@gmail.com), залинковать сюда же в этот файл, наполнить начальной структурой. Параллельно у меня и Cursor должен быть доступ. Возможна интеграция со связкой Apps Script ↔ Supabase для автоматической Metrics-вкладки.
**Статус:** Отложено. Вспоминать о ней Claude при удобных моментах в работе.
